# 原仓库精华保留清单

上游仓库：`ExplosiveCoderflome/AI-Novel-Writing-Assistant`

## 核心判断

原项目的精华不是单次生成正文，而是一个 AI Native 长篇小说生产系统：从灵感、导演开书、世界观、角色、卷/章规划、章节执行、审校修复到知识回灌，形成可持续生产整本书的闭环。

## 必须保留的能力

1. 自动导演开书
   - 从一句灵感进入项目设定、标题组、故事宏观规划、世界与角色准备。
   - MCP 化后应体现为 Claude 代用户下达导演指令，AI Novel 后端沉淀项目状态。

2. 整本生产主链
   - 不是单章孤立生成，而是卷战略、卷骨架、节奏拆章、章节执行和批次恢复。
   - 章节 brief、scene cards、质量闸门、摘要和事实账本必须持续更新。

3. 知识库与 RAG 回灌
   - 素材、拆书、世界设定、角色关系、章节摘要、伏笔和时间线要进入后续生成上下文。
   - 本地 SQLite 和导出 Markdown 是资产，不是临时缓存。

4. 写法引擎
   - 保留风格样本、特征池、反 AI 规则、类型套路和读者预期。
   - 对标作品研究要变成可复用材料，而不是聊天记录。

5. 质量修复链
   - 审校、正文修改、结构化 fallback、JSON 修复、错误原因暴露和任务恢复都属于核心能力。
   - 生成慢或失败时先查模型路由、后台日志、结构化输出和 provider，而不是让 Claude 手写正文。

6. GUI 的可见状态感
   - 原 GUI 的价值包括侧边栏、任务中心、运行态和沉浸式创作界面。
   - MCP 化后需要用状态面板、样本文档、任务清单和本地导出弥补可见性。

## 不能丢的产品精神

- 帮新手从一句灵感走到完整小说。
- 让系统记住作品，而不是每次重新解释。
- 让导演链路推进整本书，而不是让用户反复手动提醒。
- 把成功作品、素材、风格和错误修复沉淀为下一次创作能力。

## 当前 MCP 改造验收口径

- Claude 不直接写生产正文。
- 章节生成必须走 AI Novel MCP 工具。
- 每次生成前先更新 chapter brief/task sheet。
- 每次生成后读取 sample 并导出本地 Markdown。
- 角色、风格、事实、伏笔和章节摘要必须能回灌。
- 模型路由要可查、可测、可替换。
- 旧路线污染要逐项清理并备份数据库。
